# UI Audit — 2026-04-09

**Scope**: Walked every page in the broker sidebar (`/dashboard`, `/search`,
`/search/[orgnr]` Oversikt tab, `/pipeline`, `/portfolio`,
`/portfolio/analytics`, `/prospecting`, `/renewals`, `/idd`, `/insurers`,
`/sla`, `/knowledge`, `/admin`) on the live staging environment using
Playwright MCP. Captured DOM snapshots, console errors, network responses,
and screenshots.

**Method**: Real browser navigation against the deployed staging frontend.
For each page: navigate, wait for content, snapshot the rendered DOM, read
console errors, and (where useful) compare against the underlying API
response. Findings ranked by severity. The "Easy-5" subset is implemented
in this same PR; the rest are documented for follow-up.

## 🔴 Critical findings

### F01 · Notifications 404 on every page (dev@local user not in DB)

**Where**: Global header notification bell, fires on every page load.
**Reproduce**:
1. Open any page on staging
2. Inspect browser console
3. See repeated `GET /bapi/notifications?unread_only=true&limit=1` → 404

**Root cause**: `api/auth.py` returns a synthetic `dev@local` user when
`AUTH_DISABLED=1` is set:
```python
if _is_auth_disabled():
    return CurrentUser(email="dev@local", name="Dev User", oid="dev-oid", firm_id=1)
```
But this user is **never inserted into the `users` table**. The notifications
router (and `saved_searches`, and any future router that does an
`azure_oid` lookup) calls `db.query(User).filter(User.azure_oid == user.oid)`,
gets `None`, and raises 404 with body `{"detail":"User record not found"}`.

**Impact**: Affects all 11 broker pages on staging. The dashboard alone fires
**12 separate 404s per load** (multiple components each making the same call).
On prod with real Azure AD this works because the `UserService.get_or_create`
flow is hit during Azure AD callback. In dev mode the `get_or_create` is
never reached.

**Fix in this PR**: Modify `api/auth.py::get_current_user` to call
`UserService.get_or_create("dev-oid", "dev@local", "Dev User")` lazily
on the first AUTH_DISABLED request. The User row is created idempotently
and all downstream `azure_oid` lookups succeed.

## 🟠 High findings

### F02 · Search results sorted alphabetically, not by relevance

**Where**: `/search` page after typing a query.
**Reproduce**:
1. Go to `/search`
2. Type `DNB` in the name field
3. Click Søk
4. Notice DNB BANK ASA (the bank a broker actually wants) appears at
   **position 14 of 20**, below "DNB BEDRIFTSIDRETTSLAG" (sports club) and
   other associations / foundations / regional units.

**Root cause**: The companies search endpoint sorts by `navn ASC`. Hits
appear in alphabetical order, which buries the actual meaningful matches.

**Fix in this PR**: Switch the search ordering to a relevance-aware
score: exact orgnr match → exact navn match → navn starts-with → navn
contains → fall back to risk_score DESC for tiebreaking. Implemented in
`api/services/company.py::list_companies`.

### F03 · Broken `Nettsted` link on company profile

**Where**: `/search/[orgnr]` Oversikt tab → Selskapsinfo card → Nettsted row.
**Reproduce**:
1. Go to `/search/984851006` (DNB Bank)
2. Click the `www.dnb.no` link in the Selskapsinfo card
3. The browser navigates to
   `https://ca-frontend-staging…/search/984851006/www.dnb.no` (404), not
   `https://www.dnb.no`

**Root cause**: BRREG returns the company website as `www.dnb.no` (no
protocol). The frontend renders it directly in `href` so the browser
treats it as a relative path.

**Fix in this PR**: Normalise the URL on the frontend before rendering —
prepend `https://` if no protocol is present. Implemented in
`frontend/src/app/search/[orgnr]/page.tsx`.

### F04 · Risk score scale mismatch (9/20 vs 0–13+ legend)

**Where**: `/search/[orgnr]` Oversikt tab → Risikoscore card.
**Symptom**: The score is displayed as `9 / 20` with a 0–20 slider, but
the legend below says `Skala: 0–3 Lav · 4–7 Moderat · 8–12 Høy · 13+ Svært høy`.
The legend implies a max of 13+ and the slider implies a max of 20.
A broker has no idea whether 9/20 means "high" or "moderate" without
reading the legend twice.

**Suggested fix**: Pick one of:
- Change the slider max to 13 (if 13+ is the real cap, anything above is
  "off the scale")
- Change the displayed denominator from `9 / 20` to just `9` and let the
  badge ("Høy") + the legend communicate the bucket
- Update the legend to span 0–20 (e.g. `0–5 Lav · 6–10 Moderat · 11–15 Høy · 16–20 Svært høy`)

This needs a product call on what the canonical scale is — the
`derive_simple_risk` function in `api/risk.py` is the source of truth.

### F05 · Prospecting "Risiko" column shows wrong scale

**Where**: `/prospecting` table.
**Symptom**: The column header is `Risiko`. The category filter buttons
above the table use the standard 0–20 buckets (`Lav 0–3`, `Moderat 4–7`,
`Høy 8–12`, `Svært høy 13+`). But the actual cell values in the column
are **72, 62, 58, 50, 45, 40, 32** — numbers far outside 0–20.

When I cross-checked the API directly:
```
GET /companies?limit=3
HELSE SØR-ØST RHF       risk_score: 3   omsetning: 108B
DNB BEDRIFTSIDRETTSLAG  risk_score: 6   omsetning: 90.6B
DNB BANK ASA            risk_score: 9   omsetning: 86.5B
```
The actual `risk_score` field uses the 0–20 scale. The prospecting page
is displaying a different metric in this column (likely a derived
"prospect score" or revenue percentile) under the wrong header. This
makes the filter buttons + column values look like two unrelated
numbers.

**Suggested fix**: Either rename the column to match the metric being
displayed (e.g. "Prospekt-score") OR change the cell renderer to show
the actual `risk_score` field. Find which one the prospecting backend is
returning by inspecting `api/services/portfolio.py` (or wherever the
prospecting list query lives).

### F06 · Insurer + product names not canonicalized

**Where**: `/portfolio/analytics` (and any other aggregate report).
**Symptom**: The `by_insurer` aggregate from `/analytics/premiums`
contains both `"Tryg Forsikring"` and `"Tryg"` as separate insurers,
same for `"Gjensidige"` vs `"Gjensidige Forsikring"`,
`"Storebrand"` vs `"Storebrand Forsikring"`,
`"Fremtind"` vs `"Fremtind Forsikring"`. Sample from the live API:

```json
"by_insurer": [
  {"insurer": "Tryg Forsikring", "count": 2, "total_premium": 4900000.0},
  {"insurer": "Tryg",            "count": 4, "total_premium":  675000.0},
  {"insurer": "Gjensidige Forsikring", "count": 1, "total_premium": 3400000.0},
  {"insurer": "Gjensidige",            "count": 7, "total_premium": 1220000.0},
  ...
]
```
Same issue with `by_product`:
- "Avbruddsforsikring" vs "Driftsavbruddsforsikring"
- "Tingsskadeforsikring" vs "Tingsforsikring"
- "Styreansvarsforsikring" vs "Styreansvar (D&O)"

**Impact**: Every aggregate report (premium share by insurer, win-loss by
insurer, product mix, commission rollup) shows the same insurer twice
with split totals. Brokers can't trust the numbers.

**Suggested fix**: Two layers:
1. **Seed data fix** — canonicalise the insurer + product names in
   `api/services/demo_seed.py` (and any other seed script) so new
   environments get clean data.
2. **Migration** — add a one-time data fix that consolidates existing
   variants in the live DBs (UPDATE policies SET insurer='Tryg Forsikring' WHERE insurer='Tryg';).
3. **Defense-in-depth** — add a `canonical_insurer_name()` helper that
   normalises strings on insert (lowercase, strip Forsikring/Forsikring AS
   suffixes, then look up against a known list).

### F07 · Stage tracker confuses ✓-marks with step ordinals

**Where**: `/search/[orgnr]` Oversikt tab → top of the page, above the
Oversikt/Økonomi/Forsikring tabs.
**Symptom**: The 7-step workflow tracker shows circles labelled
`✓ Datainnhenting`, `✓ Risikovurdering`, `3 Behovsanalyse`,
`4 Tilbud innhentet`, `✓ Tilbudsanalyse`, `6 Presentasjon`, `7 Kontrakt`.

At first glance the numbers `3`, `4`, `6`, `7` look like task counts —
"there are 4 things to do under Tilbud innhentet". They're not — they're
just the step ordinal because that step isn't done yet. The mix of ✓
(meaning done) and the step number (meaning step-N-not-done-yet) is
confusing because both are sitting in the same circle slot.

**Suggested fix**: Use one consistent visual language. Either:
- All circles always show their ordinal (1–7), and a separate ✓ overlay
  appears on completed ones
- Completed circles show ✓ (no number), incomplete circles show an
  empty/outlined circle (no number), and the step name below is enough

## 🟢 Low findings

### F14 · Staging cold-start jank

**Where**: First page load on staging (after KEDA scales the API to 0).
**Symptom**: The first browser request to `/bapi/dashboard`,
`/bapi/companies`, etc returns 500 or 503 because the Next.js rewrite
times out before the API container finishes its cold start. SWR retries
automatically and the page recovers within 5–10 seconds.

**Why this is Low**: The page eventually populates correctly, the SWR
retry hides it from the user, and **prod is not affected** (prod has
`min-replicas: 1`, staging has `min-replicas: 0`).

**Suggested fix**: Bump staging `min-replicas` from 0 to 1
(`az containerapp update --min-replicas 1`). Costs ~10 USD/month for the
extra always-on replica. Trade-off: no cold-start jank vs slightly
higher staging bill.

### F15 · Pipeline card badges unlabeled

**Where**: `/pipeline` deal cards.
**Symptom**: Each card shows `1d` and `#2` badges with no label or
tooltip explaining what they mean. Best guess: `1d` is "1 day in this
stage" and `#2` is "2 activities logged" — but a new broker has to
guess.

**Suggested fix**: Add a tooltip on hover, or replace with full labels
("1 dag i fasen", "2 aktiviteter").

### F16 · Search has no "recent" or "recommended" companies

**Where**: `/search` initial state.
**Symptom**: Until the broker types something, the page shows only the
search form. There's no list of "recently viewed", "in your portfolio",
or "high-risk in your industries" to give the broker something to click
on.

**Suggested fix**: Add a "Nylig sett" section under the form showing the
last 5 companies the broker viewed (track via localStorage or a
`user_recent_companies` table).

## ⚪ Architectural notes

### F17 · `/notifications` endpoint called multiple times per page

The dashboard fires **12 separate calls** to `/bapi/notifications?unread_only=true&limit=1`
per load. This suggests:
- React StrictMode double-rendering in development
- Multiple components calling the same SWR hook independently instead of
  sharing one cache key
- Or a useEffect with a missing dependency array

The right fix is to centralise notifications in a single
`useNotifications()` SWR hook that all components share. Once the
dev-user 404 fix in F01 lands, this becomes a perf concern instead of an
error spam concern, but it's still worth doing.

### F18 · External resources from openstreetmap.org and unpkg.com

The company profile loads:
- Map tiles from `https://*.tile.openstreetmap.org/...`
- Leaflet marker icons from `https://unpkg.com/leaflet@1.9.4/dist/images/...`

These are fine for now but worth knowing for two reasons:
1. **Privacy**: every company profile view leaks the broker's IP +
   company orgnr (in URL referer) to OpenStreetMap.
2. **CSP**: when we eventually add a Content-Security-Policy header,
   these origins need to be in `img-src`.

**Suggested fix when the time comes**: self-host the marker icons in
`frontend/public/`, and either accept the OSM dependency or proxy tiles
through the backend.

### F19 · Workflow stage numbers look like task counts

Same problem as F07 but worth flagging architecturally: showing a number
in a place users expect a count is a UX trap. Future workflow components
should use icons or empty/filled shapes instead of bare numbers.
